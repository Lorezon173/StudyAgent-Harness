## 1. 原始诉求（用户原话）

> 从零开始重构整个项目，项目的主要目的是搭建一个用于辅助学习的多 Agent，使用费曼学习法的教学逻辑，带有 RAG 本地知识库，且能够依据本地数据和历史长期记忆进行回答，持续针对性回答不同用户在不同话题不同学习进度的情况下的学习问题，并且能够高效教学（自动触发提问追问、混淆纠正、深度教学等）；设计一套体系化的量化评估标准，实现对每个部件进行性能监测和量化，以便对 Agents 进行客观合理评估以及未来可能的技术选型升级参考。

---

## 2. 规模评估与子系统识别

诉求跨越至少 6 个相对独立子系统：

| 子系统 | 核心问题 |
|---|---|
| A. 教学方法学引擎 | 费曼学习法的状态机/教学循环 |
| B. RAG 本地知识库 | 索引、检索、证据门控、增量更新 |
| C. 长期记忆 + 用户画像 | 跨会话个性化、学习进度追踪、知识图谱化 |
| D. 自动触发器 | 追问/纠错/深度教学的判定逻辑（事件驱动） |
| E. 多 Agent 编排 | 角色划分、调度、状态汇聚 |
| F. 量化评估体系 | 每部件的指标定义、采集、监测、对比基准 |

**识别出的关键矛盾**：现有项目用苏格拉底教学法（AI 是引导者），用户要求费曼法（AI 是受众/教练），两者循环主体相反。此矛盾通过"融合式动态切换"方案解决。

## 4. 架构方案对比（三选一）

### 🅐 LangGraph 单图 + 事件 Hook（保守型）
- 一个主图承载所有节点；事件退化为状态字段
- 优势：LangGraph 工具链全保留；劣势：事件驱动名存实亡

### 🅑 自研 Agent Runtime + 显式 Event Bus（彻底型）
- 抛弃 LangGraph，asyncio + Pub/Sub
- 优势：事件驱动彻底；劣势：失去工具链，TDD 难

### 🅒 LangGraph 骨架 + 外置事件总线（混合型）★ 选定
- 主图负责稳定流水线，协作环内部事件驱动
- 优势：稳定入口/出口 + 灵活中段；保留 checkpoint/流式/回放；评估友好
- 劣势：两套范式需边界文档

**决策：🅒 混合型**

## 5. 设计评审中的两点重要反馈（用户提出）

### 反馈 1：评估体系需要两层
用户指出 Critic 只评估用户掌握度和检索置信度不够，需要一个庞大的评估体系评估**整个 Agent 集群所有成员**的量化指标，推动集群升级与客观衡量产品性能。

→ 解决：明确切分两层
- **L1 在线评估**：Critic Agent，会话内运行，影响下一步动作
- **L2 旁路评估**：EvalKernel，独立运行，评估全集群 + 全部件 + 系统级，输出选型建议

### 反馈 2：Orchestrator 纯规则会导致教学退化
用户质疑：纯固定流程规划，用户不按既有流程走时教学效果差，引入 LLM 决策（升级为独立 Agent）能否动态调整阶段。

→ 解决：Orchestrator 升级为「规则优先 + LLM Conductor 兜底」
- Agent 数量 4 → 5（新增 Conductor）
- 规则引擎覆盖高频路径（~1ms，零成本）
- 规则未覆盖时召唤 Conductor（LLM 决策，处理长尾）
- 形成 L2→L1 反馈环：Conductor 高频触发模式 → EvalKernel 提议新规则

# 方案呈现规范（引用即解释）

> 本规范约束 Claude **沟通侧**的行为：给用户任何方案、计划、设计时，如何呈现才能让用户在不熟悉底层产物的情况下也能判断方案优劣。
> 与项目 CLAUDE.md 里约束**探索侧**的「引用即承诺读」对称——那条管"我写文档前必须真读源码"，这条管"我给方案时必须把引用的产物解释清楚"。

## 核心原则：引用即解释

方案里出现的**每一个**脚本、文档、hook、skill、模块、配置等产物，都欠用户一句交代：

> 它**是什么** / 归**哪个子系统** / 现在**干嘛** / 从**哪触发或进入** / 在**哪**（`file:line`）。

不允许把只有自己读过的产物当成你我共享的词汇直接甩出来。对用户而言，未经解释的产物名就是无法展开的黑盒 token；而方案的每个决策又往往建立在这些产物之上，用户会被迫要么盲信、要么迷路，无法判断优劣。

红线信号（出现即停下补解释）：
- 方案里写了某个脚本/文档名，但全文没有一句说明它是什么、在哪。
- 用"沿用现有的 X"「调用 Y」这类说法，而 X / Y 从未向用户介绍过。
- 自己心里清楚某产物的作用，默认用户也清楚。

## 默认开头两件套

任何方案的**开头**先给这两样，再展开正文：

### 1. 资产清单

把方案会碰到的每个产物列成一行，字段固定：

| 名字 | 归属子系统 | 功能（干嘛） | 触发 / 入口 | 位置（file:line） |
|---|---|---|---|---|

这本质是项目 CLAUDE.md「抽屉清单法」用在解释侧——让用户先拿到地图，再看方案才不悬空。
**诚实标注**：自己真读过实现的，照实填；只引用了名字没读实现的，在功能列标 `⏳ 未读实现` 并尽快补，不许凭空脑补功能描述（沿用「引用即读」精神）。

### 2. 三色血缘

资产清单每一行（或单独一栏）打标，标出该方案对它的影响：

- 🟢 **保持现状** — 既有产物，方案只是用它，不改。
- 🟡 **改动** — 既有产物，方案会修改它。
- 🔴 **新建** — 方案要新造的产物。

让用户一眼看出影响半径，并把「既有的」和「我要新造的」分开——这两者混在一起最容易让人懵。

## 默认随决策给判据（不只给结论）

方案正文里**每一个有取舍的决策**——选型、"用 A 不用 B"、"沿用现有而非新建"、路径/结构/命名的取舍——都必须默认附上判据，不允许只甩结论让用户反问"为什么不那样"。对应项目方法论的「对照路径法」：任何机制都有 2 列对照表，空格不可存在。

**判据的标准形态：两列对照表**

| | 本方案（选它） | 备选（没选） |
|---|---|---|
| 做法 | … | … |
| 代价 / 收益 | … | … |

表后补一句**选择理由**：为什么本方案的收益压过备选。让用户有支点判断优劣，而非被迫盲信或反问。

**何时必须给**：方案里出现"用 / 采用 / 沿用 / 改成 / 放在 X"这类**选择动词**、且存在合理备选时——出现即欠一份判据。红线信号：有选择动词、无判据 = 这段没写完。

**何时可省（克制）**：唯一解、无合理备选的机械步骤、用户已明确指定的选项。不对鸡毛蒜皮强行对照——避免判据泛滥淹没真正的关键决策（呼应「宪法克制」与「简单优先」）。轻量场景下两列对照表可压缩成一句"选 A 不选 B，因为 Y（B 的代价是 Z）"。

## 按需手段（用户觉得某处可疑时再展开）

- **分层可展开**：先给一个不懂细节也能判断"形状对不对"的概览，细节默认收起；用户点哪条决策可疑，再把那一条展开到源码级（`file:line`）。

## 资产清单活文档

每个项目维护**一份** master 活文档：`docs/asset-inventory.md`。

- 它是该项目资产的持续累积索引，**随时可查**，由 Claude **随时维护更新**。
- 每次方案产出时：方案正文里附**那次的**资产清单 + 三色 diff（聚焦本次涉及的产物，不重复抄全量）；同时把新出现 / 有变更的资产同步进 master 活文档。
- master 只存**项目事实**（本项目具体有哪些产物）。"默认做清单+三色"这条**跨项目规矩**属于工作方式，不写进 master，由本规范承载。
- 单向晋升红线：master 是项目内产物，**绝不**把某项目的资产内容上灌到全局。

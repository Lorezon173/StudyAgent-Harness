# StudyAgent 全局开发规范

本规范从全局 CLAUDE.md 同步而来，作为项目级强制约束。

## 语言要求

- 始终使用中文回答问题和进行沟通

## 命令执行规范

- 安装库或运行耗时命令时，禁止使用 sleep 强制等待
- 改用轮询方式：每隔 30 秒主动执行检查命令（如查看进程状态、检查文件生成等），确认进展后再决定下一步操作
- 目标是通过主动检查而非被动等待来加快整体进展

## README 维护规范

- 每次完成一个任务后，必须检查并更新当前项目根目录的 README 文档
- 更新内容应包括：新增/变更的功能、架构调整、API 变化、使用方式变更等
- 确保 README 始终反映项目的最新状态，避免文档与代码脱节

## 模块开发规划要求

所有模块的编程，都必须遵循**三层详细规划**，禁止直接动手写代码：

1. **模块总览** — 明确整个模块的目标边界、技术选型、依赖关系、与现有架构的衔接方式
2. **子模块概述** — 将总览拆解为子模块，逐一说明每个子模块的职责、接口契约、数据流、状态管理、错误处理策略
3. **子模块详细实施计划** — 针对每个子模块，以文字描述说明：
   - 需要哪些函数，每个函数的作用、输入与输出
   - 需要完成哪些功能
   - 字段与变量如何设计
   - 关键数据结构

   **注意：详细实施计划中不要直接给出代码，而是用文字描述清楚"做什么"和"怎么设计"。**

规划必须经用户确认后，方可进入编码阶段。未经规划直接编码视为违规。

## 规划产出物管理

当使用 superpower 的 brainstorming 技能进行规划时：

- **原始计划文档作为 raw 材料保存** — brainstorming 产生的完整计划文档，必须保存到项目根目录下的 `superpowers/` 文件夹中，按模块命名归档，用于留存和回溯
- **spec 设计来源必须追溯至 raw 文件** — 后续编写的 spec（如 `specs/` 目录下的规范文件），其设计内容必须来源于 `superpowers/` 文件夹中对应模块的详细设计文档，确保 spec 与规划一致、有据可查
